#+TITLE: Rooted Bytestring Conversion Bindings Slice
#+DATE: 2026-07-21
#+OPTIONS: toc:nil

* Outcome

This slice binds Gerbil's official =:std/misc/bytes= hexadecimal conversion
surface without returning an unrooted, newly allocated =scheme-object= to
Rust.

The accepted Rust APIs are:

- =SchemeBytevector::to_bytestring=
- =GerbilRuntime::bytevector_from_bytestring=
- =RootedSchemeString=
- =RootedSchemeBytevector=
- =BytestringDelimiter::{SPACE, Compact, Character}=

The =bytevector= spellings are official aliases of the =u8vector= functions,
so one binding implementation supports all four names:

- =u8vector->bytestring= / =bytevector->bytestring=
- =bytestring->u8vector= / =bytestring->bytevector=

* Official Gerbil semantics

The implementation owner is =:std/misc/bytes= in the tracked Gerbil checkout.
It establishes these rules:

- hexadecimal digits are uppercase;
- the default delimiter is one space;
- =#f= selects compact output;
- a delimiter may be any Unicode scalar character;
- delimited input must contain exactly two hexadecimal digits per byte and the
  selected delimiter between adjacent bytes;
- malformed input raises a Gerbil error.

The native bridge detects conversion failures and maps them to root token zero.
The checked Rust ABI maps zero to =GerbilStatus::InvalidValue=.

The embedded archive intentionally links only the binding module and Gambit,
not the complete Gerbil std module graph. Therefore =scheme/native.ss= carries
a small AOT bridge implementation that pins the official
=:std/misc/bytes= wire semantics. It does not move conversion into Rust, and it
does not depend on unresolved std/prelude procedures such as =__with-catch=.
Validation is branch-based and returns token zero instead of raising across the
C ABI. The lowered =scheme/generated/native.scm= remains tracked and drift
checked.

* Root ownership contract

Conversion allocates a Scheme string or u8vector. Returning its raw Gambit word
would not keep it reachable after the FFI call, so this slice adds a module-owned
root table.

1. Scheme converts and stores the new object in the root table.
2. The C ABI returns a positive signed 64-bit =GerbilRootId=.
3. Rust accessors address the value only through that root id.
4. =RootedSchemeString= or =RootedSchemeBytevector= owns exactly one root.
5. =Drop= calls =gerbil_scheme_rust_root_release= on the runtime owner thread.
6. Zero, stale, wrong-type, and already released roots fail closed.

The rooted wrappers are neither =Clone= nor =Copy=. Their lifetime is tied to
=GerbilRuntime=, so safe Rust cannot finalize the runtime while roots remain.

* C ABI

The public v1 header now declares:

- =GerbilRootId=
- =gerbil_scheme_rust_bytevector_to_bytestring_root=
- =gerbil_scheme_rust_bytestring_to_bytevector_root=
- rooted string length and character projection;
- rooted bytevector length and byte projection;
- =gerbil_scheme_rust_root_release=.

The same header update closes existing declaration drift for the void and
read-only bytevector functions added by the preceding slices. The ABI major
version remains =1=: the project has not released v1, and these declarations
complete that version rather than creating a second ABI.

* Scenario and performance contract

=bytevector-bytestring-round-trip= executes two warm paths:

- five-byte Scheme bytevector to spaced uppercase bytestring, including root,
  Rust copy, and release;
- spaced bytestring to Scheme bytevector, including root, Rust copy, and
  release.

Each path records median and p95 in nanoseconds. Both median budgets are below
=500us= and both p95 budgets are below =900us=. The total scenario, including
runtime initialization, must remain below the harness hard maximum of =500ms=.

These are hard sub-millisecond operation gates, not millisecond-only display
values. The receipt keeps exact =*_ns= observations so later bulk-copy or root
table optimizations are directly comparable.

The first local AOT receipt is:

| operation | median | p95 |
|-----------+--------+-----|
| bytevector to bytestring | 4717ns | 13595ns |
| bytestring to bytevector | 2467ns | 5215ns |

The complete measured scenario took =34212125ns=.

* Generated SCM provenance boundary

The committed =scheme/generated/native.scm= is an AOT build artifact, not a
Rust source-analysis input.  The bindings crates therefore do not depend on a
Gerbil parser, Rowan, or =gerbil-scheme-language-project-harness=.

The CI matrix originally compared regenerated SCM byte for byte after only
normalizing the compiler timestamp.  Gerbil installations built from the same
revision can allocate different numeric suffixes to internal compiler gensyms,
so that check rejected semantically equivalent output on both Linux/source and
macOS/Homebrew runners.  Parsing Scheme inside the bindings repository would
put language-tooling policy on the wrong side of the C ABI boundary.

The replacement contract is syntax independent:

1. =GERBIL_SCHEME_RUST_UPDATE_GENERATED_SCM=1= writes the compiler output and
   a provenance header together.
2. The header binds the artifact to SHA-256 fingerprints of =build.ss=,
   =gerbil.pkg=, =scheme/native.ss=, and =scheme/native.ssi=.
3. A second SHA-256 fingerprint covers the committed SCM body, so manual body
   drift fails closed.
4. =GERBIL_SCHEME_RUST_CHECK_GENERATED_SCM=1= validates both fingerprints and
   selects the committed SCM as the actual Gambit compile input.
5. All matrix lanes still compile, link, and execute the C ABI; none parse
   Scheme syntax or normalize compiler identifiers.

The language project harness remains an upstream development and policy tool.
It may test Gerbil language semantics, but it is not a dependency of the
runtime bindings or their generated-artifact provenance gate.
