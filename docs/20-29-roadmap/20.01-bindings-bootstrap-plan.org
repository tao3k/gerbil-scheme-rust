* 20.01.00 Plan Contract

This plan implements [[file:../10-19-design/10.01-gerbil-scheme-rust-bindings-rfc.org][GSR-RFC-0001]].

Work advances in checked vertical slices.  A phase is complete only when its
declared commands or artifacts produce a recorded receipt.  File presence,
checkboxes, and successful code generation are not substitutes for build and
runtime evidence.

* 20.01.10 Current Snapshot

As of 2026-07-18 the independent repository boundary is operational:

- the dual-license, Cargo workspace, Gerbil package, canonical =build.ss=,
  Bzlmod graph, Rust toolchain, and Linux/macOS CI exist;
- =gerbil-scheme-sys= owns ABI version 1 and the public C header contract;
- =gerbil-scheme= owns the exclusive, thread-affine safe runtime and scalar
  projection surface;
- Cargo and Bazel consume the canonical Gerbil AOT archive;
- statistical Scenarios distinguish C control, raw Gerbil AOT, and safe Rust
  wrapper cost in nanoseconds per operation;
- Marlin extraction has completed inventory and classification, but Marlin does
  not yet consume this repository as a pinned dependency.

Phase 0 is complete.  Phases 1, 3, and 4 have working vertical slices.  Phase 2
remains open for module invocation and owned value handles.  Phase 5 is the
active cross-repository migration.

* 20.01.20 Delivery Rules

- Keep the repository bindings-only.
- Preserve Cargo, =build.ss=, and Bazel ownership boundaries.
- Prefer the smallest end-to-end Gerbil/Rust slice over a broad module copy.
- Do not import Marlin business schemas as temporary dependencies.
- Keep unsafe code localized and reviewed.
- Do not claim real-runtime support from mock or serialized-fixture tests.
- Record exact commands, toolchain identities, and artifact paths for every
  acceptance gate.

* 20.01.100 Immediate Next Slice

The active slice makes this repository the single upstream bindings dependency:

1. parameterize archive/link/header/compiler helpers in
   =gerbil-scheme-native-build=;
2. port generic Marlin build-support tests as provenance-recorded Scenarios;
3. publish a pinned git revision for downstream validation without declaring a
   release;
4. change Marlin native build support to consume that pinned dependency;
5. retain Marlin IR, policy, Deck, workspace, and CLI wrappers downstream;
6. delete only the duplicate generic implementation after both repositories are
   green.

No Marlin domain type, schema identifier, package identity, or adapter module
may enter the upstream bindings public API.

* 20.01.110 Risk Register

| Risk | Early signal | Control |
|------+--------------+---------|
| Gerbil native ABI varies by toolchain | layout or symbol drift across CI lanes | ABI fingerprint, pinned versions, fail-closed validation |
| Bazel and Cargo build different surfaces | tests pass in only one entry point | shared declared inputs and parity gate |
| =build.ss= logic is duplicated in Bazel | command or artifact drift | keep =build.ss= canonical and test Bazel delegation |
| Marlin semantics leak into bindings | public names or dependencies mention Marlin domains | dependency deny gate and API review |
| unsafe lifetime contract is underspecified | retained pointer or callback failures | explicit ownership table and sanitizer-oriented tests |
| dual-license metadata drifts | package/header/license mismatch | automated SPDX and package metadata check |
| CI is green only with warm local state | clean checkout or cold cache failure | cold-cache gate and recorded artifact paths |

* 20.01.120 Implementation Snapshot

** 2026-07-17 Native Runtime Completion

- Canonical =build.ss= output is packaged into a Cargo-linked static archive.
- Gambit =___setup=, generated module linker, and =___cleanup= are owned by a
  minimal C lifecycle shim.
- =GerbilRuntime= is exclusive, thread-affine, ABI-checked, and safe to call.

- A second live runtime owner is rejected; cleanup permanently finalizes the
  process runtime and later initialization is rejected safely.
- Cargo format, tests, and Clippy are green.
- Bazel serializes the native toolchain and runs a consolidated Cargo gate plus
  the canonical Gerbil gate.

* 20.01.130 Remaining Gaps

The bootstrap no longer has an in-process runtime gap.  Remaining work belongs
to API expansion: exception-to-status containment, owned value handles with
explicit release functions, and additional ABI-versioned scalar/string calls.

- Complete module load, function invocation, and owned value handle ABI v1.
- Decide whether process-based =gxi= invocation is a core feature or a separate
  feature from embedded/native binding.
- Define the initial ownership table for values, handles, callbacks, and
  buffers.
- Add callback/panic containment and invalid native caller Scenarios.
- Record provenance for every generic helper rewritten from Marlin.
- Make Marlin consume the pinned upstream dependency and remove duplicates.
- Establish explicit cold/warm CI baselines and ratio variance gates.
